# commit スキル 補足手順(エッジケース)

発生頻度は低いが、発生した際に必要な詳細手順をまとめる。該当状況になったときだけ参照する。

## リモート同期時にuntrackedファイルがpullと衝突する場合

Step 1.5 の同期手順中、untrackedファイルがリモートと衝突して `git pull` が失敗した場合の対処。

1. 衝突している当該ファイル・ディレクトリをバックアップする
2. 当該ファイル・ディレクトリを削除する
3. `git pull` を再実行する
4. pull完了後、バックアップの内容を確認し、必要な変更を手動でマージする

## 変更を複数ブランチ(複数PR)に振り分ける場合の注意

Step 3 で分割した単位をそれぞれ別ブランチにコミットして別PRにする場合、対象外ファイルを退避する手段として `git stash push -- <pathspec>` を使わない。この操作は指定ファイルの差分だけを退避するわけではなく、ワーキングツリー全体の状態を1つのstashとしてスナップショットしたうえで、pathspecに一致するファイルのみをワーキングツリー上でHEAD状態に戻す（他のファイルはそのまま残る）。そのため、このstashを別ブランチでpopすると、退避時点の全ファイルの差分が復元されてしまい、他の単位の変更まで意図せず混入する。

代わりに以下の手順を使う。

1. 元のブランチ（全単位の変更が未コミットで存在する状態）で、1つ目の単位のファイルのみ `git add` してコミットする
2. 分岐元のベースブランチに `git switch` で戻る（他単位の未コミット変更はそのまま working tree に残る。1つ目の単位のファイルはコミット時点のHEAD内容に戻る）
3. `git switch -c <branch2>` で2つ目のブランチを作成する
4. 1つ目の単位のファイルが（別ブランチでコミット済みの内容と同一の差分で）再度 modified 表示される場合は `git restore <file>` でHEAD状態に戻し、2つ目の単位のファイルのみが残っていることを `git status --short` で確認する
5. 2つ目の単位のファイルを `git add` してコミットする

## ファイルのリネーム・削除・移動を含む変更をステージする際の参照確認

Step 4 でステージ内容にファイルのリネーム・削除・移動が含まれる場合、旧パス・旧名称への参照が残っていないかをgrepで全件確認する。

確認対象はドキュメント（README・docs配下・ガイド類）、skill/agent定義内の相互参照（`Skill X をロードする` など名前で呼ばれている箇所）、issue/PRテンプレート、CIワークフロー、設定ファイル内のパス・キー指定。1層だけ確認して終わらせず、未更新の参照は修正して同じコミットに含める。
