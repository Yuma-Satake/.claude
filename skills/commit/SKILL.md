# commit

引数: $ARGUMENTS

## 手順

### Step 1: Git状態の把握

1. `git status` でステージ状態と変更ファイルを確認する
2. `git diff HEAD` で変更の差分を確認する
3. `git log --oneline -5` で直近コミット履歴を確認する
4. `git branch --show-current` で現在のブランチを確認する

### Step 1.5: リモートとの同期確認

`git status` でローカルがリモートより遅れている（behind）場合は、コミット前に以下の手順で同期する。

1. ローカルに未コミットの変更がある場合: `git stash` で一時退避する
2. `git pull` でリモートの変更を取り込む
3. stash した場合: `git stash pop` で変更を復元する
4. コンフリクトが発生した場合: 内容を確認して手動で解消し、`git add <file>` でマークしてから Step 3 へ進む
5. untrackedファイルがリモートと衝突して pull が失敗した場合: `reference.md` の該当手順に従う
6. `git pull` が "divergent branches" エラーで失敗する場合（ローカル・リモート双方に新規コミットがある分岐状態）: ローカルの未pushコミットがリモート側の変更と同一ファイルを変更していないか `git log --oneline origin/<branch>` で確認し、衝突しないと判断できれば `git pull --rebase` で統合する。ローカルコミットが既に他者と共有され得る状態、または衝突の可能性がある場合はユーザーに手動でのreconcile方法（merge/rebase）を確認する

リモートより進んでいる（ahead）だけの場合はこのステップをスキップする。

### Step 2: ブランチの準備

引数に `-b` が含まれていない場合はこのステップをスキップし Step 3 へ。

1. 会話の文脈に issue 等のブランチ名情報がある場合はそのブランチ名を使用する
2. 不明な場合は差分の内容に基づいて適切なブランチ名を生成する
3. `git switch -c <branch-name>` でブランチを作成する

### Step 3: コミットの分割判断

変更が複数の論理的な単位に分けられる場合は分割してコミットする。1コミット = 1つの論理的な変更単位が原則。

1. Step 1 で確認した差分を分析し、論理的に独立した変更のまとまりを特定する
2. まとまりが1つであれば Step 4 へ。複数ある場合は全まとまりに対して Step 4 を繰り返し、全て完了したら Step 5 へ進む

例: スキーマ変更・バックエンドロジック・APIルート・UI の変更が混在する場合は、それぞれ別コミットに分割する

独立した複数の新規ファイル（例: 複数のスキル定義・複数のエージェント定義）を同一目的（例: 特定システム連携のためのスキル群を一括整備する）のもとでまとめて追加した場合は、ファイル同士に機能的な依存がなくても1コミットにまとめてよい。目的や対象が異なる複数の変更が混在する場合は、その単位ごとに分割する。

分割単位をそれぞれ別ブランチ（別PR）にコミットする場合は `reference.md` の手順に従う。対象外ファイルの退避に `git stash push -- <pathspec>` を使うと他単位の変更が混入するため使わないこと。

### Step 4: コミットの実行

1. 対象ファイルを `git add <files>` でステージする
   - `git add -A` は `.env` などのセンシティブファイルや無関係の変更を誤ってコミットするリスクがあるため使用しない
   - Step 3 で分割コミットする場合、`git add <files>` で指定していないファイルも、以前の操作で既にインデックスにステージ済みだと一緒にコミットされてしまう。`git commit` の直前に必ず `git status` でステージ内容を確認し、意図した論理単位のみが含まれているかを確認する
   - ステージ内容にファイルのリネーム・削除・移動が含まれる場合は `reference.md` の確認手順に従い、旧パス・旧名称への参照が残っていないか確認する
2. 下記のコミットメッセージルールに従ってメッセージを生成する
3. `git commit -m "<message>"` でコミットする

#### コミットメッセージ形式

`<type>: <subject>` の形式で記載する。

| type | 用途 |
| --- | --- |
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `refactor` | 機能変更を伴わないコード改善 |
| `style` | フォーマット変更（動作に影響なし） |
| `docs` | ドキュメントのみの変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルド・ツール関連 |

- type: 変更内容の種類を表すキーワード
- subject: 日本語で記載

例: `fix: ログイン時のバリデーションエラーを修正`

コミットメッセージは差分の内容を最優先に生成する。会話の文脈はあくまで補足情報として扱い、メッセージの根拠は常に実際のコード変更に基づくこと。

### Step 5: プッシュ

引数に `-p` が含まれていない場合はこのステップをスキップして終了する。

1. `git ls-remote --heads origin <branch>` でリモートにブランチが存在するか確認する
2. リモートにブランチが存在する場合: `git push` を実行する
3. リモートにブランチが存在しない場合: `git push -u origin <branch>` を実行する
