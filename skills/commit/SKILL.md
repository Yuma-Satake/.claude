# commit

引数: $ARGUMENTS

## 現在のGit状態

- デフォルトブランチ: !`git remote show origin | grep HEAD`
- 現在のブランチ: !`git branch --show-current`
- ステータス: !`git status`
- 変更の差分: !`git diff HEAD`
- 直近コミット: !`git log --oneline -5`

## 動作モード

- 引数なし → 現在のブランチにコミットする。ブランチ作成・pushは行わない
- `-b` → 新規ブランチを作成してからコミットする
  - 会話の文脈にissue等のブランチ名情報がある場合はそのブランチ名を使用
  - 不明な場合は差分の内容に基づいて適切なブランチ名を生成
  - ブランチ作成には `git switch -c` を使用
- `-p` → コミット後にリモートへpushする
  - 現在のブランチに対して `git push` を実行する
  - リモートにブランチが存在しない場合は `git push -u origin <branch>` を使用する
- `-b -p` → 新規ブランチを作成し、コミット後にpushまで行う

## コミットルール

### 分割

- 1コミット = 1つの論理的な変更単位
- レビュワーが「このコミットは何をしているのか」を容易に理解できる単位にする
- 大きな変更は論理的な単位（スキーマ、バックエンド、APIルート、UIなど）でコミットを分割する

### メッセージ形式

`<type>: <subject>` の形式で記載する。

| type | 用途 |
| --- | --- |
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `refactor` | 機能変更を伴わないコード改善 |
| `style` | フォーマット変更（動作に影響なし） |
| `docs` | ドキュメントのみの変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルド・ツール関連 |

- type: 変更内容の種類を表すキーワード
- subject: 日本語で記載

例: `fix: ログイン時のバリデーションエラーを修正`

### メッセージ生成の優先度

コミットメッセージは**差分の内容を最優先**に生成する。会話の文脈はあくまで補足情報として扱い、メッセージの根拠は常に実際のコード変更に基づくこと。

- 差分で実際に変更されたファイル・コード内容を分析し、その変更が「何をしているか」を正確に記述する
- 会話の中で議論された意図や背景は参考にしてよいが、差分に反映されていない内容をメッセージに含めない

## 基本ルール

- ブランチ操作には `git checkout` ではなく `git switch` を使用する
- `-p` オプションが指定されていない場合、pushは行わない
