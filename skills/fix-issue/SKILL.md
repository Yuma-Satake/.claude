# fix-issue

引数: $ARGUMENTS

- `$0`: issue番号（コマンド実行に使用）
- `$0` 以降に文字列が続く場合、それはユーザからの追加指示であり、プランニング・実装時に考慮する

GitHub issue $0 をコーディング規約に従って修正する。

## 現在のGit状態

- 現在のブランチ: !`git branch --show-current`
- リポジトリ: !`gh repo view --json nameWithOwner -q .nameWithOwner`

## issueの内容

`gh issue view $0 --comments`

- issueが見つからない場合には、ユーザに確認を取ること
- フォークしたリポジトリで作業している場合には、親リポジトリのissueを確認すること

## 手順

### 1. ブランチ作成

`feature/issue-$0` ブランチを作成する。

### 2. issueについての情報収集

issueの本文・コメントを精読し、以下の関連情報を収集する。

- issue内で言及されているPR・issue（`#xxx` 形式）があれば `gh pr view` / `gh issue view` で内容を確認する
- 外部リンクがあれば WebSearch / WebFetch で情報を取得を試みる

### 3. コード/ライブラリについての情報収集

- Exploreエージェントを用いて、修正対象のコードや類似するコードについての情報を収集する
- ライブラリの仕様やバージョンに関わる情報が必要な場合には、 Context7 MCPを用いて情報を収集する

### 4. プランニング

1. 収集した情報を元に、対応方針を検討し、それをPlanとしてまとめる
2. Planに抜け漏れがないかセルフレビューを行い、必要に応じて修正する
3. Planをユーザに提示し、内容に問題がないか確認し、承認が得られるまで修正を繰り返す
4. ユーザの承認が得られたら、合意した対応方針をissueにコメントとして記録する

- Planを作成する場合には、ステップ7でレビューをどのサブエージェントに依頼するべきかを検討し、記載する
- issueへのコメントには、対応方針の要約と決定事項を簡潔にまとめること
- プランニング中にユーザとの会話で方針変更や追加の決定があった場合、それらもコメントに含めること

### 5. 実装

収集した情報・作成したPlanを元に実装を行う。
コミットなどは行わない。

### 6. 品質チェック

- type-check・lintを実行し、エラーがないことを確認する
- それ以外にもテストがあれば実行し、エラーが発生しないことを確認する

### 7. レビュー

適切なサブエージェントに変更内容のレビューを依頼し、指摘があれば修正する。

- 指摘がなくなるまでレビュー→修正のサイクルを繰り返す
