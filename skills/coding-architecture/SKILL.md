# アーキテクチャに関する規約

コードの記述後に、以下の前提を確認した上で、変更及びそれらに関連する（ここでいう"関連"は単に近しい場所にあることだけではなく、同様の概念を実装している他ファイルなども含める）コードを観察し、規約をチェックリストとして使用することで、設計におけるアンチパターンを排除できるものである。

## 前提

以下の規約は、全てのコードに対して適用される概念である。
レイヤードアーキテクチャや、クリーンアーキテクチャなどの違いはあるが、責務の分離や重複したコードの排除など、普遍的な共通の概念については同じであり、その部分について記述する。

最も重要なことは **既存のコードからアーキテクチャを観察し、そのアーキテクチャの目的に沿ったコードの配置/分離を行うこと** である。
コードの1行についても **「なぜここに記述するべきなのか」「この変更を加えるためには、この箇所を変更すべきなのか、新しく追加するべきなのか」** を考えることが重要である。

## 規約

### 関心の分離/依存についての観点

関心の分離及び依存の方向についての観点としては以下を意識する必要がある。
その上でアンチパターンと解消方法を提示する。

- 関心の分離：コードはそれが属するドメインにまとまりの中、かつ、そのコードが属する正しいレイヤー（UI層、ドメイン層、インフラ層）に配置する
- 依存について：ドメイン同士/レイヤー同士は可能な限り疎結合であるべきである。かつ、相互に依存する場合には、依存が一方向になるように設計する

| アンチパターン | 検出の観点 | 解消方法 |
| --- | --- | --- |
| ドメインロジックの漏出 | UI層・ハンドラ層に業務ルールの判定・計算・分岐が書かれている | ロジックをドメイン層に移動し、UI層は表示と入力の変換に専念させる |
| ドメイン層の上位層依存 | ドメイン層がUIの表示都合やフレームワークの型をimportしている | ドメイン層はプレーンな型のみで定義し、表示用の型はUI層で定義する |
| インフラ都合の型の貫通 | DBスキーマやAPIレスポンスの型がそのままUI層まで到達している | 層の境界でドメインモデルへ変換し、外部都合の型を内側に持ち込まない |
| feature間の直接依存 | あるfeatureが別のfeatureの内部実装を直接importしている | 共有すべき概念を共通層に昇格させる |
| プリミティブ型の引き回し | 業務上の意味を持つ値がstringやnumberのまま複数の層を通過している | 値オブジェクトやドメインオブジェクトを定義し、検証とロジックをその型に集約する |
| booleanフラグによる状態表現 | 複数のbooleanの組み合わせで状態を表現し、あり得ない組み合わせが型上存在する | union型やenumで状態を直接表現し、不正な状態を代数的な型表現で排除する |
| 万能モジュールへの集約 | utilsやcommonに関心事の異なる処理が無秩序に追加されている | 処理が属する関心事を特定し、その責務を持つモジュールに配置する |

### DRY原則の観点

DRY原則についての観点としては以下を意識する必要がある

- 重複の対象：DRYが排除すべきなのはコードの字面の重複ではなく、知識（仕様・ルール）の重複である。同一の知識はコードベース上の一箇所に定義する
- 共通化の判断：「同じ理由で変更されるか」を基準とし、変更理由が同じものだけを共通化する。字面が似ているだけで変更理由が異なるコードは共通化しない

| アンチパターン | 検出の観点 | 解消方法 |
| --- | --- | --- |
| 知識の重複 | 同一の業務ルールや計算が複数箇所に別々に実装されている | 一箇所に定義を集約し、他はそれを参照する |
| 偶然の一致の共通化 | たまたま似ているだけで変更理由が異なるコードが共通化されている | 共通化を解消して別々に定義し、それぞれの変更理由に沿って独立に進化させる |
| フラグ引数による分岐の増殖 | 共通関数が呼び出し元ごとのフラグ引数で内部分岐している | 関数を呼び出し元の関心事ごとに分割し、共通部分だけを抽出する |
| 既存実装の再発明 | 既存の共通関数や既存パターンを把握せず同じ処理を新規実装している | 変更前に類似の既存実装を検索し、既存のパターンに揃えて再利用する |
| マジックナンバー・定数の散在 | 同じ意味を持つリテラル値が複数箇所に直書きされている | 意味を表す名前を付けた定数として一元管理する |
| 早すぎる抽象化 | 1〜2箇所の利用のために汎用的な抽象レイヤーが導入されている | 具体的な実装のまま重複を許容し、3回目の利用で共通化を判断する |
