# knowledge-review

会話の最初から現在までを振り返り、再利用可能な知見を抽出して適切な場所への反映を提案する。ファイルを先に眺めて「該当がないか探す」のではなく、必ず会話の振り返りから始めること。

## Step 1: 知見候補の全列挙

この段階では汎用性を判断せず、機械的に拾い上げることに専念する。以下のシグナルごとに該当の有無を1つずつ明示し（該当がない場合も「該当なし」と記す）、該当する出来事を列挙する。

1. ユーザーから受けた指摘・修正・却下（特に同種の指示を2回以上受けたもの）
2. ユーザーが表明した新しい指示・好み（「今後は〜して」のような将来に向けた要望を含む）
3. コマンドやツールの実行が失敗し、別のやり方でリトライした箇所
4. skill の手順どおりに進めて詰まった箇所、手順に書かれていない判断を自分で行った箇所
5. 事前に知っていれば手戻りを防げた発見（環境・ツール・ライブラリの仕様など）
6. 期待した結果と実際の結果が食い違い、原因調査が必要になった箇所
7. うまく機能した工夫・手順で、次回も再現する価値があるもの

各候補は「何が起きたか」「そこから得られる教訓」の1〜2行で書き出す。全シグナルが該当なしの場合はこのスキルを終了する。

## Step 2: 汎用性の判定と反映先の振り分け

各候補に対して次の判定を順に適用する。

1. 一過性の判定: その日の障害・環境の一時的な不調・二度と発生しない状況に起因する知見は反映しない
2. 反映価値の判定: 「次回以降の作業でこの記述を読んだとき、行動が変わるか」で判断する。既に解決済みの個別事象の詳細（修正したバグの原因、そのとき限りの調査結果など）は、記録しても次回の行動を変えないため反映しない。同じ手戻りが再発しうる知見のみ次の判定に進める
3. 既知の判定: 反映先の候補（ユーザーレベルの rules・skills、対象プロジェクトの CLAUDE.md や .claude/ 配下）に同じ内容が既に正しく書かれているなら反映しない。既に書かれているが記述が不正確・不足している場合は「既存記述の修正候補」として次の判定に進む
   - 既存の rules・skills の記述どおりに作業して失敗した場合は、特に重要な修正候補として扱う。記述が「常に成り立つ」と断定していて、実際には環境やプロジェクトによって結果が異なると判明したなら、断定を確認手順や条件分岐に置き換える修正を汎用候補として挙げる（例: 「事前ビルドは不要」という断定 → 「デプロイスクリプトにビルドが含まれるか確認し、含まれない場合は先にビルドする」）。プロジェクト側への例外注記だけで済ませると、別のプロジェクトで同じ失敗が再発する
4. 汎用性の判定: 別のリポジトリ・別のプロジェクトで同じ作業をしても、この知見がそのまま成立するかを判定する
   - 特定のプロジェクト名・社内サービス名・リポジトリ内のファイルパスなどの固有名詞を使わないと教訓を記述できないならプロジェクト固有
   - 固有名詞なしで書ける場合でも、組織・チーム固有の運用ルール（コミットメッセージへのチケットID付与など）はプロジェクト固有として扱う
   - ツールに関する知見は「ツール自体の一般的な性質」（例: git stash は untracked ファイルを対象にしない）なら汎用、「そのプロジェクトでのツールの運用」（例: このリポジトリでは pnpm を使う）ならプロジェクト固有
   - 迷った場合は「6ヶ月後に無関係なプロジェクトでこの記述を読んで役に立つか」で判断し、それでも迷うならプロジェクト固有として扱う
5. 抽象化の検討: プロジェクト固有と判定した候補についても、一段抽象化すると汎用的な教訓にならないかを検討する（例: 「このリポジトリは pnpm を使う」という固有の知見 → 「パッケージを追加する前に lockfile の種類でパッケージマネージャを確認する」という汎用の手順）。抽出できた教訓は独立した候補として追加し、判定1から同じようにかけ直す。プロジェクト固有の失敗は汎用手順の確認不足を露呈していることが多く、固有の知見をプロジェクトに記録するだけではその改善機会を逃す
6. 反映先の決定:
   - 汎用かつ特定の作業手順に関するもの: 該当する `~/.claude/skills/` のスキル
   - 汎用かつ作業全般で常に守るべき制約・スタイル: `~/.claude/rules/`（適切なファイルがなければ新規作成を提案する）
   - プロジェクト固有: 対象プロジェクトの CLAUDE.md（なければ `.claude/` 配下の適切なファイル、それもなければ CLAUDE.md の新規作成を提案する）。ユーザーレベルには入れない
   - 作業対象が `~/.claude` 自体である場合は、rules・skills 本体への反映として扱う

## Step 3: ユーザー確認と反映

1. `AskUserQuestion` ツールで確認を取る。採用候補ごとに「反映先ファイル・変更内容の概要・汎用/プロジェクト固有の判定理由」を提示する。Step 2 で見送った候補も同じタイミングで理由付きで報告し、ユーザーが判定を覆せるようにする。候補が多く1回の質問に収まらない場合は複数回に分けて確認する
2. 承諾された候補のみファイルを作成・更新する。既存記述の削除・書き換えも、承諾がない限り行わない

## Step 4: 反映内容のコミットとpush

Step 3 でファイルを作成・更新した場合、コミットとpushはユーザーに確認せずそのまま実行する。

1. 変更が行われたリポジトリ（`~/.claude` や対象プロジェクト）ごとに、反映で変更したファイルのみを `git add <files>` でステージする
2. `chore: <変更内容の要約>` の形式で日本語のコミットメッセージを作成し `git commit` する
3. `git push` を実行する。pushに失敗する場合（リモート未設定・対象ブランチがリモートに存在しない等）はコミットのみ行い、その旨をユーザーに報告する
4. 複数のリポジトリにまたがって反映した場合は、リポジトリごとに1〜3を繰り返す

## 完了処理: 保留フラグの解除

このスキルがどのステップで終了した場合でも（Step 1 で該当なしと判定して終了した場合、Step 3 で候補が全て見送られた場合を含む）、最後に必ず対象リポジトリの knowledge-review 保留フラグを解除する。

`~/.claude/hooks/knowledge-review-flag.sh clear "$(git rev-parse --show-toplevel)" "$CLAUDE_CODE_SESSION_ID"` を実行する。commit スキル経由で呼び出されていない場合（フラグが存在しない場合）もこのコマンドは安全に失敗せず終了する。
