# knowledge-review

会話の最初から現在までを振り返り、再利用可能な知見を抽出して適切な場所への反映を提案する。ファイルを先に眺めて「該当がないか探す」のではなく、必ず会話の振り返りから始めること。

## Step 1: 知見候補の全列挙

この段階では汎用性を判断せず、機械的に拾い上げることに専念する。以下のシグナルごとに該当の有無を1つずつ明示し（該当がない場合も「該当なし」と記す）、該当する出来事を列挙する。

1. ユーザーから受けた指摘・修正・却下（特に同種の指示を2回以上受けたもの）
2. ユーザーが表明した新しい指示・好み（「今後は〜して」のような将来に向けた要望を含む）
3. コマンドやツールの実行が失敗し、別のやり方でリトライした箇所
4. skill の手順どおりに進めて詰まった箇所、手順に書かれていない判断を自分で行った箇所
5. 事前に知っていれば手戻りを防げた発見（環境・ツール・ライブラリの仕様など）
6. 期待した結果と実際の結果が食い違い、原因調査が必要になった箇所
7. うまく機能した工夫・手順で、次回も再現する価値があるもの

各候補は「何が起きたか」「そこから得られる教訓」の1〜2行で書き出す。全シグナルが該当なしの場合はこのスキルを終了する。

## Step 2: 汎用性の判定と反映先の振り分け

各候補に対して次の判定を順に適用する。

1. 一過性の判定: その日の障害・環境の一時的な不調・二度と発生しない状況に起因する知見は反映しない
2. 既知の判定: 反映先の候補（ユーザーレベルの rules・skills、対象プロジェクトの CLAUDE.md や .claude/ 配下）に同じ内容が既に正しく書かれているなら反映しない。既に書かれているが記述が不正確・不足している場合は「既存記述の修正候補」として次の判定に進む
3. 汎用性の判定: 別のリポジトリ・別のプロジェクトで同じ作業をしても、この知見がそのまま成立するかを判定する
   - 特定のプロジェクト名・社内サービス名・リポジトリ内のファイルパスなどの固有名詞を使わないと教訓を記述できないならプロジェクト固有
   - 固有名詞なしで書ける場合でも、組織・チーム固有の運用ルール（コミットメッセージへのチケットID付与など）はプロジェクト固有として扱う
   - ツールに関する知見は「ツール自体の一般的な性質」（例: git stash は untracked ファイルを対象にしない）なら汎用、「そのプロジェクトでのツールの運用」（例: このリポジトリでは pnpm を使う）ならプロジェクト固有
   - 迷った場合は「6ヶ月後に無関係なプロジェクトでこの記述を読んで役に立つか」で判断し、それでも迷うならプロジェクト固有として扱う
4. 反映先の決定:
   - 汎用かつ特定の作業手順に関するもの: 該当する `~/.claude/skills/` のスキル
   - 汎用かつ作業全般で常に守るべき制約・スタイル: `~/.claude/rules/`（適切なファイルがなければ新規作成を提案する）
   - プロジェクト固有: 対象プロジェクトの CLAUDE.md（なければ `.claude/` 配下の適切なファイル、それもなければ CLAUDE.md の新規作成を提案する）。ユーザーレベルには入れない
   - 作業対象が `~/.claude` 自体である場合は、rules・skills 本体への反映として扱う

## Step 3: ユーザー確認と反映

1. `AskUserQuestion` ツールで確認を取る。採用候補ごとに「反映先ファイル・変更内容の概要・汎用/プロジェクト固有の判定理由」を提示する。Step 2 で見送った候補も同じタイミングで理由付きで報告し、ユーザーが判定を覆せるようにする。候補が多く1回の質問に収まらない場合は複数回に分けて確認する
2. 承諾された候補のみファイルを作成・更新する。既存記述の削除・書き換えも、承諾がない限り行わない
