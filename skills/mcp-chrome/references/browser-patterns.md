# ブラウザ操作パターン集

Chrome MCP でのブラウザ操作における具体的なテクニック・回避策をまとめたリファレンス。フォーム入力・値取得・DOM 操作・ダイアログ回避など、サイト横断で使える汎用パターン。

## フォーム操作

### form_input の挙動

- `form_input` はフィールドの既存値を完全に置き換える。事前に `triple_click` でクリアする必要はない
- あるフィールドの入力が他フィールドを自動補完する場合（例: 郵便番号→住所）、補完後に `form_input` で上書きすれば問題なく動作する
- 同じ構造のフォームを複数回操作する場合、`read_page` で取得した `ref` はページ再訪後も同番号になることが多く、再利用できる

### form_input が効かないフィールド

SPA の combobox・カスタムドロップダウン・ラジオボタン等は `form_input` で値をセットできないことがある。

回避策: `ref` 指定の `left_click` → `type` → `key(Enter/Tab)` で入力する。

### テキストの上書き

- テキスト入力の上書きには `triple_click` で全選択してから `type` で入力する
- ボタンクリックが反応しない場合は `find` でボタン要素を特定し、`ref` 指定でクリックする

### type 直後の click は反映前の古い状態を送信するリスクがある

`computer` の `type` で入力した直後に、確認なしで検索ボタンなどを `click` すると、入力が実際にはフィールドに反映されておらず、空欄または前回値のまま送信されてしまうことがある（JASRAC/NexTone/RIAJ の検索フォーム等で複数回発生）。

回避策: `type` の後に `screenshot` を撮って入力欄に狙った値が実際に入っていることを確認してから、次のクリック操作に進む。特に同じフォームで複数回検索し直す場合（値を変えて再検索する等）は、前回の値がクリアされずに残っていたり、逆に入力済みのはずが空になっていたりするケースがあるため、送信前の目視確認を省略しない。

## 値の取得

- 画面上で切れている値は `read_page` で `ref_id` を指定し、DOM 内の要素から全文を取得できる
- クリップボード API はセキュリティ制約で使えない場合がある。`read_page` による DOM 直接参照を優先する
- コピーボタンのクリック + `navigator.clipboard.readText()` は信頼性が低い

## テキスト入力の注意点

- `type` アクションは絵文字（✅ など）を入力できない場合がある
- 絵文字入力が必要な場合は、セルや編集可能要素をダブルクリックして編集モードに入った上で `javascript_tool` の `document.execCommand('insertText', false, '絵文字')` を使う

## JavaScript DOM 操作

- `javascript_tool` で要素を検索して `.click()` を実行する場合、DOM ツリーの親要素をたどる回数（`parentElement` の繰り返し）が正しいか慎重に確認する。ページの DOM 構造が想定と異なると、意図しない要素がクリックされるリスクがある
- `.click()` を実行する前に、対象要素のテキストやクラス名をログ出力して正しい要素であることを検証する
  ```js
  JSON.stringify({ text: targetBtn.textContent, class: targetBtn.className })
  ```
- 検証に失敗したらクリックせずエラーを返す。**先に検証、後にクリック** を守る
- `scrollIntoView()` はページのスクロールコンテナが `window` 以外（例: 固定レイアウトの内部コンテナ）の場合に機能しないことがある。スクリーンショットで要素が画面に表示されていることを確認してからクリックする
- `find` や `read_page` で見つからないボタン（`<span>` ベースのカスタムボタン等）は、アクセシビリティツリーに出ないことがある。その場合は `javascript_tool` でセレクタ指定して操作する
- `javascript_tool` で jQuery のイベントハンドラなどを `toString()` で読み出そうとすると、その中身に Cookie やクエリ文字列らしき文字列（`?xxx=yyy` や `document.cookie` 等のパターン）が含まれる場合、結果が `[BLOCKED: Cookie/query string data]` として返されずブロックされることがある。これはツール側のセキュリティ機構であり、マスキング等で回避を試みるのではなく、別の手段（該当 JS ファイルを `curl` 等で直接取得して静的に読む、ネットワークタブで実際のリクエスト先を確認する等）に切り替える

## リンク・ナビゲーションの罠

- `href="#"` のリンクは、実際には JavaScript の `click` イベントハンドラで `window.open()` を呼び出し、別タブでページを開く実装になっていることがある。クリック後に同一タブの見た目が変化しなくても、それだけでは「反応していない」と判断せず、`tabs_context_mcp` でタブ一覧を再取得し、新規タブが開いていないか確認する

## 確認ダイアログ（alert / confirm / prompt）

JavaScript の `alert` / `confirm` / `prompt` やブラウザネイティブの確認ダイアログが開くと、後続の MCP コマンドが **Detached while handling command** で止まる。

対策:
1. `confirm()` を使うボタンはクリックしない。フォームの場合は `fetch` で POST して dialog を回避する
2. どうしてもクリックが必要なら、事前にユーザーに「ダイアログが出たら手動で OK を押してほしい」と依頼する
3. 既に開いてしまった場合は、ユーザーに手動 dismissal を依頼してセッションを復帰する

## スクリーンショット座標とビューポート座標の変換

- ビューポートサイズとスクリーンショットサイズは異なる場合がある（例: viewport 2042x1124, screenshot 1476x812）
- `javascript_tool` で取得した `getBoundingClientRect()` の座標はビューポート座標なので、`computer` ツールでクリックする場合はスクリーンショット座標に変換が必要: `ss_x = vp_x * ss_width / vp_width`
- 変換後の座標が画面外（スクリーンショットの幅/高さを超える）の場合、要素がビューポート外にあるため、先にスクロールが必要

## browser_batch の制約

- `browser_batch` の中で `read_page` は使えない（`unknown tool` エラーになる）。navigate → read_page の流れは、2ステップを別々の呼び出しに分ける必要がある
- 複数の `browser_batch` を同じタブIDで並列送信するとタブが上書きされる。別々のページを同時に操作したい場合は `tabs_create_mcp` で新規タブを作ってから、それぞれ異なるタブIDを指定する

## form_input での日本語入力時の文字化け

- `form_input` で日本語のテキスト（住所・氏名など）を入力すると、まれに一部の文字が別の文字に化けることがある（例:「下落合」→「下落午」、「宛」→「宿」）。ログ上は成功したように表示されるため、入力後は必ず `zoom` 等で実際の表示テキストを確認し、化けていた場合は同じ `ref` に対して再度 `form_input` で正しい値を入力し直す

## select 要素の value 属性と表示テキストのずれ

- ネイティブの `<select>` 要素に `form_input` で値をセットする際、渡した値が option の `value` 属性ではなく表示テキストに対してマッチしているように見えるケースがある（例: 月選択で value="6" のoptionが表示テキスト"7"だが、value に"6"を渡すと表示テキスト"6"のoption（value="5"）が選択される）。number 系の option value と表示テキストがズレている select では、渡す値と選ばれる表示が直感と異なることがあるため、選択後は必ずスクリーンショットで実際の表示値を確認し、意図した値と一致するまで調整する

## ネイティブ select 要素の選択

- ネイティブの `<select>` 要素は、座標クリックしてもドロップダウンの選択肢がスクリーンショットに映らないことがある（OSネイティブUIとして描画されるため）
- `computer` ツールで要素をクリックしてフォーカスした状態で、目的の選択肢の先頭文字を1回キー入力すると、ブラウザ標準の type-ahead 機能で選択が切り替わる（例: 選択肢が A/MX/CNAME/TXT/AAAA/NS/SRV のとき、AAAAを選ぶには `a` を1回押す。先頭文字が重複する選択肢がある場合は複数回押すことで巡回する）
- Enterキーで確定しようとすると、select自体ではなく囲んでいる `<form>` がsubmitされてしまうことがある。値変更後は他のフィールドをクリックしてblurさせ、送信は別途フォームの送信ボタンで行う

## クリック後の画面遷移確認

- クリックした直後に画面が遷移したかどうかを、続けて別の操作を送る前に必ずスクリーンショットやタイトルで確認する。特に以下のケースでは、見た目上クリックが成功しているように見えても実際には遷移していないことがある
  - テキスト入力直後に同じ操作の流れで送信ボタンをクリックした場合（1回目のクリックが入力欄のフォーカス解除だけに使われ、実際の送信は2回目のクリックで初めて成立することがある）
  - ページ遷移直後に `find` で取得した要素参照（ref）をすぐクリックした場合（要素がまだインタラクティブになっていない、または参照が古い可能性がある）
- 遷移していないと判明したら、同じ操作（同じ座標/refへのクリック、または `find` の再実行→クリック）をもう一度試す

## ユーザーによる同時操作の可能性

- Claude in Chrome はユーザーが普段使っているブラウザを共有するため、自分が操作していない間にユーザー自身が同じタブを手動操作し、画面が遷移していることがある。想定していない画面遷移や入力値の変化に気づいた場合、自分の直前の操作が誤動作したと決めつけず、ユーザーが操作した可能性も踏まえて現在の状態を確認し、必要ならユーザーに状況を尋ねる

## その他のブラウザ操作テクニック

- ページの読み込み完了はタブのタイトル変化で判断できる
- 小さい UI 要素は `zoom` で拡大して確認してから操作する
- ページ遷移が遅いサイトでは、操作後に `wait` で数秒待機してから `screenshot` を撮る
