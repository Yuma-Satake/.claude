# agent-use

複数のサブエージェントを並列駆動する場合の原則。

## 起動前の設計

- タスクの分解を行い、依存関係を整理して並列実行可能な箇所がある場合には、積極的に複数のエージェントで並行して作業を進める
- 複雑なシステムをレビューする際は、観点を分割して複数の code-reviewer サブエージェントを並列起動する（例: 整合性レビュー・ループ安定化レビューを別々のエージェントに担当させる）。1体に全観点を委ねるより指摘の質・網羅性が高くなる
- WebFetch を使った調査は、サブエージェントではなく直接 WebFetch ツールを複数並列呼び出しする方式を優先する。WebFetch サブエージェントを多数同時起動するとレート制限（429）で一斉失敗するリスクがある

## プロンプトの書き方

- サブエージェントのプロンプトには「このタスクの調査・実行にあたって追加のエージェントを起動しないこと」と明記する。サブエージェントが内部でさらにエージェントを起動すると制御不能に膨らむ。指示された状況以外でのみ省略可
- サブエージェントに成果物ファイルを書かせる場合、analysis・report・findings・summary 等のファイル名は Write ツールのガード（サブエージェントはレポートファイルではなく本文で報告せよという制約）に拒否される。ガードの回避を指示するのではなく、レポート系と解釈されない名前（process-log.md、proposals.md など）を最初から指定する
- 同一タスクを複数のサブエージェント（code-reviewer等）に分担させる際、「対応不要と判断した事項」「ユーザー確認済みの経緯」などの共有情報は、1体のプロンプトにだけ書いて済ませない。全サブエージェントのプロンプトに同じ内容を明記する。1体にだけ伝えて他に伝えなかった場合、伝えていないサブエージェントが同じ事項を独自に再指摘してくる

## 隔離とworktree

- ファイルを変更するエージェントはAgentツールの `isolation: "worktree"` を指定して隔離する。読み取りのみのエージェントは隔離不要。不要な隔離を付与すると、プロンプト内で既存の別worktreeパスへのアクセス・移動を指示した際にBashツールがそのworktreeに対して一時的に使用不能になり、エージェントがEnterWorktreeで自分のworktreeに再切替する回復操作を挟むはめになることがある
- 書き込み先が scratchpad などリポジトリ外のディレクトリに限定されるエージェントも隔離不要。worktree はリポジトリ内ファイルの競合を防ぐための仕組みであり、リポジトリを変更しないエージェントに適用しても無駄なオーバーヘッドになる
- worktree内で作業するエージェントは最初に `git fetch origin` を実行する。ブランチは `origin/<デフォルトブランチ>` または `origin/<feature-branch>` を起点にする。ローカルブランチを起点にしない（worktreeはローカルの状態を共有しないため）

## 共有リソースの扱い

- 共有リソース（GitHub Issueのラベル・タスクチェックボックス、同名ブランチへのpushなど）への書き込みは1エージェントに集約する。複数エージェントから同じリソースを更新しない
- ファイル番号や採番（ADR連番など）が必要な場合、push直前に `origin` 上の状態を再確認して重複していれば振り直す
