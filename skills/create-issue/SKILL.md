# create-issue

引数: $ARGUMENTS

ユーザの要望 `$ARGUMENTS` をもとにGitHub issueを作成する。

## 現在のリポジトリ

- リポジトリ: !`gh repo view --json nameWithOwner -q .nameWithOwner 2>/dev/null || echo "不明"`
- issueテンプレート: !`ls .github/ISSUE_TEMPLATE/ 2>/dev/null || echo "なし"`

## このスキルの責務

このスキルは「**何をするか・なぜするか**」を明確にすることに集中する。
実装方針（どうやるか）の決定は `fix-issue` スキルの責務であり、このスキルには含めない。

コードベースの調査は「実装方針を決めるため」ではなく、「**ユーザへの質問の精度を上げるため**」に行う。

## 手順

### 1. コードベースの調査

ユーザの要望に関連するコード・ファイル・機能を **Exploreエージェント** で調査する。

調査で把握すること：
- 要望に関連する既存のコード・ファイル・コンポーネントはどこか
- 技術的に実現できるか、制約や前提条件はあるか
- 類似の既存実装やパターンはあるか

### 2. 要件のヒアリング（繰り返し）

調査結果を踏まえて、**AskUserQuestionTool** で要件を詰める。

ヒアリングのポイント：
- 1回あたり最大4問まで
- ユーザの回答を受けて、不明点や矛盾があれば追加で質問する
- 以下が揃ったらヒアリングを終了する

揃えたい情報：
- **目的・背景**: なぜこれが必要か、どんな問題を解決するか
- **対象・スコープ**: 何を変える/追加する/修正するか（影響範囲）
- **受け入れ条件**: どうなっていたら完了か（具体的な完了の定義）
- **制約・前提**: 既存の動作を壊してはいけない点、依存関係など
- **非機能要件**: パフォーマンス・セキュリティ・スケーラビリティへの影響があるか
- **リスク**: ユーザーデータ・既存機能・外部連携などへの潜在的な影響はあるか

バグ報告の場合は追加で：
- 再現手順
- 期待する動作 / 実際の動作

調査で技術的な懸念点（実現が難しい可能性、影響範囲が広い箇所など）が見つかった場合は、ユーザにその点を共有して要件に反映する。

### 3. スコープの評価とUnit分解

ヒアリング完了後、要望の規模を評価する。

以下の条件に該当する場合、独立して実装・デプロイ可能な複数のUnit（作業単位）への分割を検討し、ユーザーに提案する：
- 受け入れ条件が5個以上になる
- 複数の独立したコンポーネントにまたがる変更が必要
- 段階的にリリースできる区切りが明確に存在する

Unit分割を提案する場合は、各Unitが「独立してデプロイ可能かつ単体で価値を持つ」ことを基準に分割案を示し、ユーザーの承認を得る。

### 4. issueの作成

ヒアリングで揃った情報をもとに `gh issue create` でissueを作成する。

issueに含める内容：
- **タイトル**: 簡潔に何をするか（言い切り形）
- **本文**: 目的・背景、対象スコープ、技術的な関連箇所（実装方針は含めない）
- **受け入れ条件**: チェックボックス形式
- **非機能要件**: 該当する場合のみ、パフォーマンス・セキュリティ等の要件
- **リスク**: 潜在的な影響がある場合のみ記載
- **関連issue**: `gh issue list` で関連する既存issueを確認し、存在すれば `Closes #xxx` / `Related to #xxx` で紐づける

issueテンプレートがある場合は `.github/ISSUE_TEMPLATE/` のフォーマットに従う。
`gh label list` でリポジトリのラベルを確認し、適切なものを付与する。

### 5. fix-issueへの引き継ぎ確認

issue作成後、作成したissue URLをユーザに伝え、「このissueを今すぐ対応しますか？」とテキストで確認する。
対応する場合は `fix-issue` スキルを作成したissue番号で呼び出す。
