# hook-create

## 実装前の実データ検証

Bashツールの `tool_input`/`tool_response` の正確なフィールド構成は、公式ドキュメントだけでは判明しないことがある。フック本体を書く前に、以下の手順で実データを確認する。

1. 一時的なロギングフック（標準入力をファイルに書き出すだけのスクリプト）を、検証したい条件（`matcher`・`if`）で `settings.json` の該当イベントに仮登録する
2. 検証対象のコマンドを、本番環境を汚さない使い捨ての場所（scratchpad配下に作った使い捨てのgitリポジトリなど）で実行し、フックを発火させる
3. 出力されたJSONを読み、必要なフィールドの実際の構造を確認する
4. 確認が終わったら一時フックとテスト用ファイル・リポジトリを削除し、判明した仕様に基づいて本番のフックを実装する

推測でフィールド名を決め打ちしない。公式ドキュメントの記載が曖昧・不足している場合は、この手順で実測してから実装する。

## Bashツールのフック連携に関する既知の仕様

- `PostToolUse` はBashコマンドがexit code 0で終了した場合のみ発火する。コマンドが非0で終了した場合は発火しないため、フック側で改めて成功判定を行う必要はない
- gitコマンドをBashツールで実行した場合、`tool_response` に `gitOperation` という構造化フィールドが入ることがある（例: `git commit` 成功時は `gitOperation.commit.sha` と `gitOperation.commit.kind: "committed"`）。公式ドキュメントには明記されていないが、stdoutの文字列解析より確実な判定手段になる
- フック入力の `cwd` は、ハーネスが追跡するセッションの作業ディレクトリを指す。同一コマンド内で `cd` した先には必ずしも追従しない（1回の観測に基づく知見のため、挙動に依存する実装をする場合は上記の実データ検証手順で都度確認すること）

## StopフックでadditionalContextにより行動を強制する場合の注意

Stopイベントは、タスクが完了して応答が終わった場合だけでなく、ユーザーへの確認待ちの質問・承認待ちで応答が終わった場合にも同様に発火する。フック側はこの2つを区別できないため、`additionalContext` で「〜を実行せよ」と行動を強制するメッセージを注入する設計にする場合、メッセージの文言自体に「直前の自分の応答がユーザーへの確認待ち・承認待ちで終わっている場合は、その行動を実行せず何もせず終了する」という条件分岐を明記する。「ユーザーから新規に依頼された作業があるか」だけを問う文言では、確認待ちの質問で終わった直後のターンが「依頼作業なし」と誤判定され、意図しない強制実行を招く

## flag+hook自動化パターンにおける自己再武装(re-arm)問題

`PostToolUse`（`git commit` 検知）でフラグを立て、`Stop` でフラグを見て自動タスクを起動する設計では、その自動タスク自身がgit commitを伴う場合（自身の出力をコミット・pushするなど）、そのcommitが同じ`PostToolUse`フックを再度発火させ、フラグを再度アームしうる。session_idの一致を前提にした解除（session_idベースのclear）は、そのcommitが別セッション（fork起動など）内で実行される場合には保証されない。

対策: フラグ setter 側で、今回のcommitで変更されたパスを `git -c core.quotePath=false diff --name-only HEAD~1 HEAD` で取得し、自動化タスク自身の既知の出力先（例: `logs/`・`knowledge/` ディレクトリ）のみで完結しているかを判定する。該当する場合はフラグを立てないガードを入れる。これによりsession_idの一致に依存せずループを防げる。`-c core.quotePath=false` は必須で、これを付けないと非ASCII（日本語など）ファイル名がダブルクォート+8進エスケープで出力され、パス接頭辞マッチが常に不一致になりガードが機能しない。

同じcommitイベントを起点に複数の独立したflag+hook自動化（例: 複数のスキルがそれぞれ別のフラグファイルを使うケース）が併存する場合、各自動化のsetterは自分のフラグしか気にしないため、一方の自動化の自己コミットがもう一方のフラグを再武装しうる（相互発火・ping-pong）。この相互作用は各設計を独立に見ているだけでは気づけないため、hookを新設する前に、同じイベント（同じcommit等）に既に登録されている他のhookが存在しないか確認する。

## hookコマンドのパス指定

`~/.claude/settings.json` のようなユーザーレベルの設定ファイルは、複数マシン間でgit等により同期・共有されることがある。hookのcommandでスクリプトパスを指定する際、`/Users/<username>/...` のような絶対パスをハードコードすると、ユーザー名やホームディレクトリ構成が異なる別マシンでフックが失敗する（`No such file or directory`）。同一ファイル内の他のhookコマンドが `~/.claude/hooks/...` の形式で統一されているなら、それに倣い `$HOME` 展開（`~/...` または `"$HOME/..."`）を使うこと。
