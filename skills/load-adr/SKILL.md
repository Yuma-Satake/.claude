# load-adr

docs/logs/ の意思決定ログから、指定されたトピックに関連する判断だけを抽出して簡潔に報告する。このスキルはフォークされたサブエージェントとして実行され、ログ本文はこのコンテキスト内でだけ読まれる。呼び出し元の会話に届くのは最終報告だけであり、報告の質と簡潔さがこのスキルの価値のすべてである。

## 入力

ARGUMENTS にトピックまたは質問が渡される（例: 「認証まわりの決定」「なぜ生成画像を保存しない方針なのか」）。ARGUMENTS が空の場合は全ログの俯瞰ダイジェストを返す。

## 手順

1. カレントプロジェクトの docs/logs/ のファイル一覧を確認する。ファイル名は YYYY-MM-DD-テーマ名.md 形式であり、ファイル名だけで対象を絞れることが多い
2. ファイル名で絞れない場合は、トピックの関連語で grep して対象ファイルを特定する。関連語は表記ゆれ（日本語/英語、略語）を考慮して複数試す
3. 対象ファイルを読む。冒頭の status と末尾の追記セクションを必ず確認する。追記で決定が覆っている場合（superseded）は最新の判断を正とし、覆った経緯を1行で添える
4. 報告をまとめる

## 報告フォーマット

- 関連する判断を1件1〜2行で書く。判断内容、理由の要点、判断ラベル（採用 / 条件付き採用 / 不採用 / 保留）、出典ファイル名を含める
- 質問形式で聞かれた場合は、冒頭に直接の答えを1〜2文で書き、根拠となる判断を続ける
- 関連する保留・未決事項は必ず含める。呼び出し元が「まだ決まっていないこと」を知らずに作業を進めると、勝手に決めてはいけないことを暗黙に決めてしまうため
- 全体で20行以内を目安にする。ログ本文の丸写しや網羅的な羅列はしない。トピックに関係しない判断は、どれほど興味深くても含めない
- 該当する判断が見つからない場合は「該当なし」と明言し、存在するログのテーマ一覧（ファイル名）を添える。誤って無関係な判断を関連があるように報告しない

## 引数なしの場合

俯瞰ダイジェストを返す。ログ1ファイルにつき1行で、テーマ、status、決定の要点を書く。詳細が必要なテーマがあればトピック指定で再実行できることを最後に添える。

## docs/logs/ が存在しない場合

このプロジェクトには意思決定ログがない旨を報告する。adr スキルで記録を始められることを添える。
