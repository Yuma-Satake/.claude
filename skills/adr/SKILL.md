# adr

議論・アイデア出し・意思決定の内容を、後から「なぜそう決めたか」「なぜやらなかったか」を辿れる形でプロジェクト内に記録する。名前はadrだが、一般的なADR（決定単位・連番・不変）より軽い形式を使い、決定単位ではなく議論単位で1ファイルにまとめる。1回の壁打ちから決定は複数出るのが普通であり、決定ごとに分割すると議論の文脈が失われるため。ユーザーが「ADR」と言った場合もこの形式で書く。

## いつ使うか

- ユーザーが「ログに残して」「記録して」「保存して」と明示したとき
- 構想の壁打ちや設計議論が節目に達したとき（方針が決まった、issue化が完了した、Planが承認された）は、こちらから「この議論をログに残しますか」と提案する。提案にとどめ、承諾なしに保存しない

## 保存先とファイル名

- 保存先はカレントプロジェクトの docs/logs/。なければ作成する
- ファイル名は YYYY-MM-DD-テーマ名.md（例: 2026-07-03-付加価値機能の構想.md）。日付接頭辞で自然に時系列に並ぶため、索引ファイルは作らない
- 書く前に docs/logs/ の既存ファイルを確認する。同じテーマの議論の続きであれば、新規ファイルを作らず既存ファイルの末尾に「## 追記 YYYY-MM-DD」セクションとして追記する。同一テーマかどうかは日付ではなく内容で判断する
- gitリポジトリの外で実行された場合は、保存先をユーザーに確認する

## テンプレート

新規ファイルは次の構成で書く。該当する内容がないセクションは省略してよい。

```markdown
# テーマ名

status: decided
date: 2026-07-03
related: #6 #7

## 背景

なぜこの議論が始まったか。1〜3文で書く。

## 出たアイデアと判断

- OGP対応: 採用。唯一の新規獲得施策のため最優先
- ソーシャルプルーフ: 条件付き採用。匿名カウントから始め、顔ぶれ表示はオプトイン
- ログイン必須化: 不採用。ゼロフリクションという現在の強みを殺すため

## 決まったこと

- 生成画像は保存せず、ベース画像と配置パラメータから再合成する

## 保留・未決

- イベントエンティティ化は需要が確認できてから判断する

## 次のアクション

- #7 から着手する
```

## 書き方のルール

- 会話の逐語録ではなく判断の要約を書く。ユーザーの発話の意図は保ち、細部の言い回しは捨てる
- 各アイデアには判断ラベル（採用 / 条件付き採用 / 不採用 / 保留）と理由を1〜2文添える。特に不採用と保留の理由は省略しない。このログの最大の価値は、後から「なぜあれをやらなかったのか」に答えられることにあり、採用案だけならissueを見れば分かる
- 言い切りの簡潔な文章で書く。"---"・"**"・絵文字は使用しない
- 冒頭のメタ情報（status / date / related）はYAMLフロントマターではなくプレーンな行で書く。人間が読むためのファイルであり、区切り記号を使わない書式に合わせるため
- status は open（未決事項が残っている）/ decided（方針確定）/ superseded（後の判断で覆った）のいずれか
- 追記で過去の決定が覆った場合、本文は書き換えずに追記セクションへ新しい判断と理由を書き、冒頭の status だけ更新する。判断の変遷自体が記録の価値であるため
- related には関連する issue / PR / ADR を書く。issue化まで終えた議論なら必ず番号で紐づける

## 保存後

保存または追記したファイルパスをユーザーに伝え、内容に過不足がないか確認する。

## 保存済みログの参照

過去のログを参照する必要があるときは、docs/logs/ を直接Readせず load-adr スキルを使う。ログが増えても会話のコンテキストを圧迫せずに要約だけを取得できる。ただし、このスキル自身が追記先を判断するために既存ファイルを確認する場合は直接読んでよい。
