# agent-creator

## agentファイルの配置

`~/.claude/agents/<name>.md` または プロジェクトの `.claude/agents/<name>.md` に配置する。

## frontmatterフィールド一覧

| フィールド | 必須 | 説明 |
|---|---|---|
| `name` | 必須 | 一意識別子。小文字とハイフンで記述する |
| `description` | 必須 | Claudeがいつこのagentを使うべきかを示すテキスト |
| `tools` | 任意 | 使用可能なツール一覧。省略時は全ツール継承。明示指定しない限り省略する |
| `disallowedTools` | 任意 | 継承ツールから除外するツール |
| `model` | 任意 | `sonnet`/`opus`/`haiku`/full model ID。省略時は親から継承。明示指定しない限り省略する |
| `permissionMode` | 任意 | `default`/`acceptEdits`/`auto`/`bypassPermissions` など |
| `maxTurns` | 任意 | agentic turnsの上限 |
| `skills` | 任意 | preloadするskill名。内容全体がcontextに注入される |
| `background` | 任意 | `true`でbackground taskとして動作 |
| `color` | 任意だが全agentに設定する | `red`/`blue`/`green`/`yellow`/`purple`/`orange`/`pink`/`cyan`。既存agentと被らない色を選ぶ |

## サブagentで使用できないツール

以下のツールはサブagentのtoolsに指定しても動作しない。

- `AskUserQuestion`
- `Agent`
- `EnterPlanMode` / `ExitPlanMode`
- `ScheduleWakeup`
- `WaitForMcpServers`

ユーザーへの確認が必要な場合は、テキスト出力として質問を記述する。

## agentとskillの使い分け

| | Agent | Skill |
|---|---|---|
| contextの独立性 | 独立（結果要約のみ親に返る） | 親sessionと共有 |
| 向いている用途 | レビュー・探索・並行実行 | 繰り返し手順・規約参照 |
| context消費 | 少ない（要約のみ） | 多い（全内容がcontextに含まれる） |

## descriptionの書き方

- 何をするagentかを1文で説明する
- 使用タイミング（「〜する場合に使用」「〜後に起動」）を明示する
- 起動してほしい条件を具体的に書く

悪い例: `コードをレビューするagent`
良い例: `テストカバレッジを分析し、テスト観点の漏れを指摘するQAエキスパート。テストファイルが存在するリポジトリでのみ使用する。`

## agentの設計原則

- ペルソナ（キャラクター・行動原則）はagentに持たせる
- 技術知識・規約はskillに持たせ、agentからSkillツールでロードする
- agentに具体的なskill名をハードコードしない（skill追加・変更のたびに更新が必要になる）
- 強調表現（「必ず」「絶対に」等）は使わない。Sonnet 4.6系ではオーバートリガーの原因になる
- `tools` は原則省略する（省略時は全ツール継承）。特定ツールのみに制限したい場合だけ明示する
- `model` は原則省略する（省略時は親から継承）。特定モデルを使いたい明確な理由がある場合だけ指定する
- `color` は全agentに設定する。既存agentと被らない色を選ぶ
- system promptは日本語で記述する

## system promptの構造

効果的なsystem promptは以下の5セクションで構成する。agentの種類（分析・生成・検証・オーケストレーション）によって内容は変わるが、構造は共通。

```markdown
あなたは[役割]に特化した[ペルソナ]です。[行動の制約があれば一文で]。

**役割:**
1. [主な責務]
2. [副次的な責務]
3. [必要に応じて追加]

**プロセス:**

1. **[ステップ名]**
   - [具体的なアクション]
   - [具体的なアクション]

2. **[ステップ名]**
   - [具体的なアクション]

**品質基準:**
- [測定可能な基準]
- [測定可能な基準]

**出力フォーマット:**

### セクション名
- `path/to/file:行番号` — 内容の説明

### サマリー
[N件。全体評価を1段落で。]

**エッジケース:**
- [発生しうるケース]: [対応方法]
- [発生しうるケース]: [対応方法]
```

### セクションの役割

- **役割**: agentが何をするか・しないかを1〜3文で定義する
- **プロセス**: 具体的なステップを順番に記述する。「分析する」ではなく「`git diff` を実行して変更ファイルを列挙する」のように行動レベルで書く
- **品質基準**: 出力の合否を判断できる測定可能な基準を列挙する
- **出力フォーマット**: 実際のMarkdownテンプレートとして定義する。「〜の形式で出力する」のような説明だけでは不十分
- **エッジケース**: 入力がない・結果が0件・結果が大量などの境界条件と対応方法を明示する
