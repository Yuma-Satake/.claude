# issue-fix

引数: $ARGUMENTS

引数の解釈:

- issue番号: 引数中の最初の数値トークン。以下 `<issue番号>` と表記する
- `-a`: 自律モードで実行する（位置は問わない）
- 上記以外の文字列: ユーザからの追加指示としてプランニング・実装時に必ず考慮する

GitHub issue `<issue番号>` を修正する。`~/.claude/rules/` 配下のルールおよびプロジェクト固有の `CLAUDE.md` に従って実装する。

## 実行モード

**通常モード**（デフォルト）: アプローチ選択・レビュー用skill選定・Plan承認をAskUserQuestionでユーザに確認しながら進める。

**自律モード**（`-a` 指定時）: プランニング・スキル選定・実装方針の決定に AskUserQuestion は用いず、AI が自ら判断して最後まで走らせる。ユーザ確認は以下の限定的なケースのみ許容する:

- issueが存在しない・不明瞭で修正内容が確定できない
- 既存のPRが存在し、そのまま継続するか新規作成か外部状態に依存する判断
- 未コミットの変更が現在のブランチにあり、破棄すると作業消失リスクがある
- reviewerの指摘内容が根本的に矛盾しており、外部知識なしに判断不能

これら以外は AI が自身の判断で決定し、判断根拠をログ（issueコメントや報告）に残すこと。

以降の手順で「通常モード:」「自律モード:」と記載のある箇所は、モードに応じて動作を切り替える。記載のない箇所は両モード共通。

## コンテキスト管理（両モード共通）

このスキルは情報収集・往復確認・レビューサイクルを多用するため、メインセッションのコンテキストを圧迫しやすい。以下を徹底し、必要な情報だけを残す。

- **大きい/多数のファイル調査はサブエージェントに委譲する**: 数百行を超えるファイルや10ファイル以上の横断調査をメインコンテキストで直接Readしない。Explore/general-purposeエージェントに「静的モデル・動的モデル・結論」を要約させ、根拠として必要な引用のみメインに持ち帰る。実装直前に編集対象ファイルそのものを読むのは問題ない。
- **AskUserQuestionの選択肢はユーザに聞く前に軽く裏取りする**: 選択肢がissueやリポジトリの実情と合っていないと自由回答での往復が発生し、そのやり取り全体がコンテキストに積み上がる。質問を投げる前にリポジトリ内・issue内を一度軽く探索し、実情に即した選択肢に絞ることで往復回数を減らす。
- **要約(コンテキスト圧縮)後は差分だけ読み直す**: 会話が要約された後、要約に含まれる既読ファイルを無条件に再Readしない。要約の記述で判断できる場合はそれを信頼し、実際に編集する範囲や要約に含まれない新規箇所のみ読み直す。

## 進捗管理

手順が多いため、TaskCreateで以下のタスクを作成し、各ステップの開始時にin_progress、完了時にcompletedへ更新する。

1. issueの情報収集
2. コード/ライブラリの情報収集
3. プランニング
4. 実装
5. 品質チェック
6. レビュー

## 手順

### 1. ブランチ作成

- 現在のブランチが `feature/issue-<issue番号>` の場合、作成をスキップしてそのまま使用する
- それ以外の場合、デフォルトブランチから `git switch -c feature/issue-<issue番号>` で新規作成する
- 現在のブランチに未コミットの変更がある場合、ユーザに扱いを確認する（退避・破棄・継続。自律モードでも例外として確認する）

### 2. issueについての情報収集

**調査フェーズは必ずサブエージェントに移譲する**。メインコンテキストで `gh pr view` / `gh issue view` / WebSearch / WebFetch を直接実行しない。

- **サブエージェント**に、issue本文・コメントの精読、言及されているPR・issue（`#xxx` 形式）の内容確認、外部リンクの情報取得を依頼する
- 複数の情報源がある場合、可能な限り並列でサブエージェントを起動する
- メインコンテキストには「関連する既存の議論・決定事項・背景情報」の要約と、根拠として必要な引用のみを持ち帰る

### 3. コード/ライブラリについての情報収集

**調査フェーズは必ずサブエージェントに移譲する**。メインコンテキストで対象コードのRead/Grepを直接実行しない。

- **Exploreエージェント**を用いて、修正対象のコード・類似実装・呼び出し元・テストの有無を調査する
  - 影響範囲が広い場合は `thoroughness: very thorough`、限定的な場合は `quick` を指定する
  - 大きな設計資料・大量のファイルを精査する必要がある場合も、メインコンテキストで直接Readせずサブエージェントに要約させる（「コンテキスト管理」参照）
- ライブラリの仕様やバージョンに関わる情報が必要な場合は、**general-purposeエージェント**に公式ドキュメントの確認（WebSearch / WebFetch）と要約を依頼する
- プロジェクトの `CLAUDE.md` やルールファイル（`rules/`, `docs/` 配下）の確認も **Exploreエージェント** に依頼し、プロジェクト固有の規約を要約させる

調査結果は以下の2つの視点で整理する：
- **静的モデル**: 関係するコンポーネント・クラス・モジュールの責務と相互関係
- **動的モデル**: 修正対象の主要な処理フロー（データの流れ・呼び出し順序）

### 4. プランニング

**Level 1: 実装アプローチの選定**

1. 収集した情報を元に、実現可能な実装アプローチを **複数案** 検討する
   - 各案のトレードオフ（変更範囲・リスク・保守性・パフォーマンスへの影響など）を評価する
   - アーキテクチャ上の判断が必要な場合は **Planエージェント** で方針を検討する
2. アプローチとレビュー用skillsを決定する
   - 通常モード: **AskUserQuestion** でアプローチ選択とレビュー時に使用するskillsを確認する
     - 質問を投げる前に、選択肢がissue・リポジトリの実情に即しているか軽く裏取りする（「コンテキスト管理」参照）。的外れな選択肢は自由回答の往復を招き、コンテキストを余計に消費する
     - アプローチ選択: 複数案を選択肢として提示する（推奨案をデフォルト選択としてtrueにする）
     - レビュー用skills: multiSelect: true で変更内容に適したskillsを提示する（推奨スキルをデフォルト選択としてtrueにする）
   - 自律モード: 以下の基準で **AI が自ら選定** する
     - アプローチ選定基準: issueの要求を満たすこと / プロジェクトの既存規約・パターンと整合すること / 変更範囲・リスクが必要最小限であること / 保守性を損なわないこと
     - レビュー用skills: 変更ファイルの言語・フレームワーク・レイヤー・変更の性質（機能追加/バグ修正/リファクタ）から関連するskillsを推論する（例: React変更なら `coding-react`, TypeScript変更なら `coding-typescript`）

**Level 2: 詳細タスクの展開**

3. 決定したアプローチを具体的な変更ファイル・タスクに分解する
   - 修正対象のスコープ（関連ファイル・影響範囲・テスト範囲）
   - エッジケース・デグレードリスクへの対応方針
4. 通常モード: Level 2 Planを以下のフォーマットでユーザに提示し、承認が得られるまで修正を繰り返す / 自律モード: ユーザ承認は不要、そのまま次へ進む

**Level 2 Planの提示フォーマット**

- 冒頭に変更ファイル一覧を列挙する（パスと種別＝新規/移動/編集のみ。詳細は書かない）
- ファイルごとに見出しを立て、変更点を **1箇条書き1変更** で言い切り形に書く
  - 複数の変更を「〜し、〜する」と1文に連結しない
  - 理由・規約参照・TODOの中身などの補足は変更点と分け、「備考:」として書く
  - 移動を伴う場合は「移動元 → 移動先」の形で1行にまとめる
- 末尾に確認方法（ビルド・テスト実行など）を独立した箇条書きとして添える

テンプレート:

```
### 変更ファイル一覧
- path/to/FileA.swift（新規）
- path/to/FileB.swift（移動）

### path/to/FileA.swift（新規）
- 変更点1
- 変更点2
- 備考: 補足があれば1行で

### path/to/FileB.swift（移動）
- 移動元 → 移動先
- 変更点1

### 確認方法
- 実行するコマンドやチェック内容
```
5. 決定事項をissueにコメントで記録する
   - 「選択したアプローチ」「その根拠（ある場合のみ）」「検討した代替案（あれば）」「主要な変更箇所」を箇条書きで、言い切り形で記載する
   - 要約や説明文は不要
   - 複数回に分けず、Plan確定後に一度だけ投稿する

### 5. 実装

Planに沿って実装する。

- 追加指示があればその内容を反映する
- コミットはこのスキル内では行わない（完了後にユーザ依頼で `commit` スキルを使用）
- 並列実行可能な変更（独立ファイル編集・調査）は積極的に並列化する

### 6. 品質チェック

変更内容に応じて以下を実行し、エラーがないことを確認する。並列実行可能なものは並列で実行する。

- **type-check / lint**: プロジェクトのスクリプトを `package.json` / `pyproject.toml` / `Makefile` 等から検出して実行する（例: `npm run typecheck`, `npm run lint`, `pnpm check`, `yarn tsc --noEmit`）
- テスト: 変更箇所に関連するテストがあれば実行する
- スクリプトが不明な場合は `CLAUDE.md` や README を参照する
- エラー発生時は根本原因を特定して修正する。`--no-verify` やチェックのスキップで回避しない

### 7. レビュー

プランニングで決定したskillごとに**reviewer agentを1体ずつ並列起動**する（同一メッセージ内で複数Agent呼び出し）。reviewer agentは1体につき最大1 skillしかロードできないため、skillの数だけreviewerを起動する。

各reviewerへの指示には以下を含める：
- レビューしてほしい変更ファイルと変更の概要
- 割り当てるskill名（reviewerがSkillツールでロードしてレビューに活用する。1体につき1 skillのみ）
- 「このタスクの調査・実行にあたって追加のエージェントを起動しないこと」

指摘への対応:

- すべてのreviewerの結果を回収し、必須修正をマージする（同一指摘は重複排除）
- 修正が必要な指摘に対応し、再度 skill ごとに reviewer agent を並列起動して再レビューを依頼する
- 指摘がなくなるまでレビュー→修正のサイクルを繰り返す
- 通常モード: 指摘内容に疑問がある場合はユーザに判断を仰ぐ
- 自律モード: 指摘への対応・「対応不要」の判断は AI が自律的に行い、判断根拠を完了時報告に含める。指摘内容がプロジェクト規約に対して根本的に矛盾しており、外部知識なしに判断不能な場合のみユーザに判断を仰ぐ
- 自律モード: 無限ループ防止のため、レビューサイクルは最大3周までとする。3周終了時点で残った指摘は完了時報告に「未対応の指摘とその判断」として明記する

## 完了時の報告

以下をユーザに報告する。

- 自律モード: 選定したアプローチと根拠
- 実装した変更の要点
- 実行した品質チェックと結果
- レビューで指摘・対応した内容（自律モードで未対応があればその判断）
- 次のアクション（`commit` → `pr-create` スキルの使用等）
