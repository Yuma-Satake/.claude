# issue-fix-auto

引数: $ARGUMENTS

- `$0`: issue番号（コマンド実行に使用）
- `$0` 以降に文字列が続く場合、ユーザからの追加指示としてプランニング・実装時に必ず考慮する

GitHub issue `$0` を修正する。`~/.claude/rules/` 配下のルールおよびプロジェクト固有の `CLAUDE.md` に従って実装する。

**このスキルは自律実行を前提とする。プランニング・スキル選定・実装方針の決定に AskUserQuestion は用いず、AI が自ら判断して最後まで走らせる。** ユーザ確認は以下の限定的なケースのみ許容する:

- issueが存在しない・不明瞭で修正内容が確定できない
- 既存のPRが存在し、そのまま継続するか新規作成か外部状態に依存する判断
- 未コミットの変更が現在のブランチにあり、破棄すると作業消失リスクがある
- reviewerの指摘内容が根本的に矛盾しており、外部知識なしに判断不能

これら以外は AI が自身の判断で決定し、判断根拠をログ（issueコメントや報告）に残すこと。

## 現在のGit状態

- デフォルトブランチ: !`git remote show origin 2>/dev/null | grep 'HEAD branch' | awk '{print $NF}'`
- 現在のブランチ: !`git branch --show-current`
- ステータス: !`git status --short`
- リポジトリ: !`gh repo view --json nameWithOwner -q .nameWithOwner 2>/dev/null || echo "不明"`

## issueの内容

!`gh issue view $0 --comments 2>/dev/null || echo "issue #$0 が見つかりません"`

- issueが見つからない場合はユーザに番号の確認を求める（自律実行の例外）
- フォークしたリポジトリで作業している場合は親リポジトリのissueを確認する
- 既にissueに対応するPR（`Fix #$0` 等）が存在しないか `gh pr list --search "Fix #$0"` で確認する。存在する場合はユーザに継続/新規を確認する（自律実行の例外）

## 進捗管理

TaskCreateで以下のタスクを作成し、各ステップの開始時にin_progress、完了時にcompletedへ更新する。

1. issueの情報収集
2. コード/ライブラリの情報収集
3. プランニング（自律決定）
4. 実装
5. 品質チェック
6. レビュー

## 手順

### 1. ブランチ作成

- 現在のブランチが `feature/issue-$0` の場合、作成をスキップしてそのまま使用する
- それ以外の場合、デフォルトブランチから `git switch -c feature/issue-$0` で新規作成する
- 現在のブランチに未コミットの変更がある場合のみ、ユーザに扱いを確認する（自律実行の例外）

### 2. issueについての情報収集

issueの本文・コメントを精読し、関連情報を並列で収集する。

- issue内で言及されているPR・issue（`#xxx` 形式）があれば `gh pr view` / `gh issue view` で内容を確認する
- 外部リンクがあれば WebSearch / WebFetch で情報取得を試みる
- 複数の情報源がある場合、可能な限り並列でツールを呼び出す

### 3. コード/ライブラリについての情報収集

- **Exploreエージェント**を用いて、修正対象のコード・類似実装・呼び出し元・テストの有無を調査する
  - 影響範囲が広い場合は `thoroughness: very thorough`、限定的な場合は `quick` を指定する
- ライブラリの仕様やバージョンに関わる情報が必要な場合は **Context7 MCP** でドキュメントを取得する
- プロジェクトの `CLAUDE.md` やルールファイル（`rules/`, `docs/` 配下）を確認し、プロジェクト固有の規約を把握する

調査結果は以下の2つの視点で整理する:
- **静的モデル**: 関係するコンポーネント・クラス・モジュールの責務と相互関係
- **動的モデル**: 修正対象の主要な処理フロー（データの流れ・呼び出し順序）

### 4. プランニング（自律決定）

**Level 1: 実装アプローチの自律選定**

1. 収集した情報を元に、実現可能な実装アプローチを内部で **複数案** 検討する
   - 各案のトレードオフ（変更範囲・リスク・保守性・パフォーマンスへの影響）を評価する
   - アーキテクチャ上の判断が必要な場合は **Planエージェント** で方針を検討する
2. 以下の基準で最適なアプローチを **AI が自ら選定** する:
   - issueの要求を満たすこと
   - プロジェクトの既存規約・パターンと整合すること
   - 変更範囲・リスクが必要最小限であること
   - 保守性を損なわないこと
3. **レビュー用skillsも AI が自律選定** する
   - 変更内容から関連するskillsを推論する（例: React変更なら `coding-react`, TypeScript変更なら `coding-typescript`、コミット前確認が必要なら `code-review` など）
   - 選定基準: 変更ファイルの言語・フレームワーク・レイヤー・変更の性質（機能追加/バグ修正/リファクタ）

**Level 2: 詳細タスクの展開**

4. 選定したアプローチを具体的な変更ファイル・タスクに分解する
   - 修正対象のスコープ（関連ファイル・影響範囲・テスト範囲）
   - エッジケース・デグレードリスクへの対応方針
5. 決定事項を **issueにコメント** で記録する
   - 「選択したアプローチ」「その根拠」「検討した代替案（あれば）」「主要な変更箇所」「レビューに用いるskills」を箇条書きで、言い切り形で記載する
   - 要約や説明文は不要
   - 一度だけ投稿する
6. ユーザ承認は不要。そのまま実装へ進む

### 5. 実装

Planに沿って実装する。

- `$0` 以降に追加指示があればその内容を反映する
- コミットはこのスキル内では行わない（完了後にユーザ依頼で `commit` スキルを使用）
- 並列実行可能な変更（独立ファイル編集・調査）は積極的に並列化する

### 6. 品質チェック

変更内容に応じて以下を実行し、エラーがないことを確認する。並列実行可能なものは並列で実行する。

- **type-check / lint**: プロジェクトのスクリプトを `package.json` / `pyproject.toml` / `Makefile` 等から検出して実行する（例: `npm run typecheck`, `npm run lint`, `pnpm check`, `yarn tsc --noEmit`）
- **テスト**: 変更箇所に関連するテストがあれば実行する
- スクリプトが不明な場合は `CLAUDE.md` や README を参照し、それでも不明な場合はコマンドを推定して試行する。試行してもわからない場合のみユーザに確認する
- エラー発生時は根本原因を特定して修正する。`--no-verify` やチェックのスキップで回避しない

### 7. レビュー

プランニングのステップ3で AI が選定したskillごとに **reviewer agentを1体ずつ並列起動** する（同一メッセージ内で複数Agent呼び出し）。reviewer agentは1体につき最大1 skillしかロードできないため、skillの数だけreviewerを起動する。

各reviewerへの指示には以下を含める:
- レビューしてほしい変更ファイルと変更の概要
- 割り当てるskill名（reviewerがSkillツールでロードしてレビューに活用する。1体につき1 skillのみ）
- 「このタスクの調査・実行にあたって追加のエージェントを起動しないこと」

指摘への対応:

- すべてのreviewerの結果を回収し、必須修正をマージする（同一指摘は重複排除）
- 修正が必要な指摘は AI が自律的に対応し、再度 skill ごとに reviewer agent を並列起動して再レビューを依頼する
- 指摘がなくなる、または残った指摘が「対応不要」と判断できる状態になるまでレビュー→修正のサイクルを繰り返す
- 判断根拠は完了時報告に含める
- 指摘内容がプロジェクト規約に対して根本的に矛盾しており、外部知識なしに判断不能な場合のみユーザに判断を仰ぐ（自律実行の例外）

無限ループ防止のため、レビューサイクルは最大3周までとする。3周終了時点で残った指摘は完了時報告に「未対応の指摘とその判断」として明記する。

## 完了時の報告

以下をユーザに報告する。

- 選定したアプローチと根拠
- 実装した変更の要点
- 実行した品質チェックと結果
- レビューで指摘・対応した内容（および未対応があればその判断）
- 次のアクション（`commit` → `pr` スキルの使用等）
