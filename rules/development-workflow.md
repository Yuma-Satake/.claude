# 開発ワークフローを実行する際に守るべきルール

- タスクの分解を行い、依存関係を整理して並列して実行可能な箇所がある場合には、積極的に複数のエージェントで並行して作業を進めること
- Planモード後の実行に関しては、可能な限りユーザとのやり取りをさけ、実行を完了まで継続すること
- クラウドサービスなどについて、プロジェクトの設定確認や、設定変更が必要な場合には、CLIやブラウザ操作を用いて積極的に自力で対応しようとすること
- 複雑なシステムをレビューする際は、観点を分割して複数の reviewer サブエージェントを並列起動する（例: 整合性レビュー・ループ安定化レビューを別々のエージェントに担当させる）。1体に全観点を委ねるより指摘の質・網羅性が高くなる


## 並列エージェント実行

複数のサブエージェントを並列駆動する場合の原則

- サブエージェントのプロンプトには「このタスクの調査・実行にあたって追加のエージェントを起動しないこと」と明記する。サブエージェントが内部でさらにエージェントを起動すると制御不能に膨らむ。指示された状況以外でのみ省略可
- WebFetch を使った調査は、サブエージェントではなく直接 WebFetch ツールを複数並列呼び出しする方式を優先する。WebFetch サブエージェントを多数同時起動するとレート制限（429）で一斉失敗するリスクがある
- ファイルを変更するエージェントはAgentツールの `isolation: "worktree"` を指定して隔離する。読み取りのみのエージェントは隔離不要
- worktree内で作業するエージェントは最初に `git fetch origin` を実行する。ブランチは `origin/<デフォルトブランチ>` または `origin/<feature-branch>` を起点にする。ローカルブランチを起点にしない（worktreeはローカルの状態を共有しないため）
- 共有リソース（GitHub Issueのラベル・タスクチェックボックス、同名ブランチへのpushなど）への書き込みは1エージェントに集約する。複数エージェントから同じリソースを更新しない
- ファイル番号や採番（ADR連番など）が必要な場合、push直前に `origin` 上の状態を再確認して重複していれば振り直す

## ファイル変更時の影響確認

ファイルの変更・削除・リネームを行う際は、次の対象に対して言及・参照がないか確認し、必要に応じて更新する。1層だけ確認して終わらせない。
skill名・agent名・パスのリネーム時は、確認漏れがそのまま実行時の参照エラーになるため、grepで全件確認してから変更する。

- ドキュメント（README・docs配下・ガイド類）
- 他のskill / agent定義ファイルでの相互参照（`Skill X をロードする`、`agents/X.md` など名前で呼ばれている箇所）
- issue / PRテンプレート、CIワークフローの参照
- 設定ファイル内のパス・キー指定
